# 1. 목적

본 절차서는 Công ty TNHH M&C ELECTRONICS VIỆT NAM (이하 M&C Electronics VINA)의 항공우주 제품에 대한 설계 및 개발(Design and Development) 프로세스를 체계적으로 기획, 실행, 관리하기 위한 절차를 규정한다.

본 절차서의 목적:
- AS9100D 8.3.1~8.3.6 요구사항에 따른 설계/개발 관리 프로세스 수립
- 다기능팀(CFT, Cross-Functional Team) 기반의 설계/개발 기획 및 단계별 게이트 관리
- 설계 검토(DR), 설계 검증(DV), 설계 확인(Validation)의 분리 실행 및 기록 체계 확립
- 핵심특성(KC, Key Characteristics) 식별 및 설계 전 단계에 걸친 관리
- DVP&R(Design Verification Plan & Report) 기반의 체계적 설계 검증/확인
- APQP(Advanced Product Quality Planning) 단계 참조 개발 프로세스 운영
- **CAR-017** (설계 검토, 검증, 유효성 확인 기록 불완전; DVP&R 없음; 구두 승인 관행) **시정조치 반영**

> **CAR-017 시정조치 사항**:
> - 근본원인: 설계 개발 단계에서 공식 검토/기록 문화 미정착, 구두 승인 관행
> - 시정조치 1: 설계 개발 절차서 신규 제정 (본 문서 QP-807)
> - 시정조치 2: 설계 검토 회의록 양식 제정 (QF-807-03)
> - 시정조치 3: 기존 진행 중 개발품 소급 기록 작성

# 2. 적용 범위

본 절차서는 M&C Electronics VINA에서 수행하는 모든 항공우주 제품 및 관련 서비스의 설계/개발 활동에 적용한다.

| 구분 | 적용 대상 |
|------|----------|
| 신규 설계 | 고객 요구에 의한 신제품/신부품 설계 및 개발 |
| 설계 변경 | 기존 제품의 설계 변경 (ECR/ECN 기반) |
| 파생 설계 | 기존 설계 기반 파생/변형 제품 개발 |
| 공정 설계 | 제조 공정, 금형/지그/치구 설계 |
| 소프트웨어 설계 | 임베디드 소프트웨어, 시험 프로그램 설계 (해당 시) |
| 설계 이관 | 고객 또는 외부 설계 결과물의 양산 이관 |

**적용 부서**: 개발(Phát triển), 생산(Sản xuất), 금형(Khuôn), 생산계획(KHSX), 품질(Chất lượng), 구매(PUR), 영업(Kinh doanh), 인사(HR_GA)

**적용 제외**: 고객이 설계를 완전히 제공하고 당사 설계 활동이 없는 단순 수탁 생산(OEM)의 경우, 해당 프로젝트의 설계/개발 기획 단계에서 적용 범위를 정의한다.

# 4. 용어 및 정의

| 용어 | 정의 |
|------|------|
| 설계/개발 (Design & Development) | 요구사항을 제품, 프로세스 또는 시스템의 규정된 특성으로 변환하는 일련의 프로세스 |
| DVP&R (Design Verification Plan & Report) | 설계 검증 계획 및 보고서 — 각 설계 요구사항에 대한 검증 방법, 판정 기준, 결과를 체계적으로 기록하는 문서 |
| APQP (Advanced Product Quality Planning) | 선행제품품질계획 — 고객 요구사항을 충족하는 제품 개발을 위한 체계적 방법론 (5단계) |
| PPAP (Production Part Approval Process) | 생산부품승인절차 — 공급자가 고객 요구사항을 충족하는 부품을 일관되게 생산할 수 있음을 입증하는 절차 |
| 설계 검증 (Design Verification) | 설계/개발 출력이 설계/개발 입력 요구사항을 충족하는지 확인하는 활동 (내부 기술 관점, "올바르게 만들었는가?") |
| 설계 확인 (Design Validation) | 최종 제품이 의도된 사용 또는 특정 용도의 요구사항을 충족하는지 확인하는 활동 (고객/사용자 관점, "올바른 것을 만들었는가?") |
| 설계 검토 (Design Review, DR) | 설계/개발의 적절성, 충분성, 효과성을 평가하기 위한 공식적이고 체계적인 검사 활동 |
| 핵심특성 (Key Characteristics, KC) | 변동이 제품의 적합성, 기능, 성능, 수명 또는 생산 가능성에 중대한 영향을 미치는 특성 |
| CFT (Cross-Functional Team) | 다기능팀 — 설계/개발 기획 시 관련 부서 전문가로 구성된 프로젝트 팀 |
| 설계 입력 (Design Input) | 설계/개발의 기초가 되는 모든 요구사항 (기능, 성능, 법규, 안전, 고객 요구 등) |
| 설계 출력 (Design Output) | 설계/개발 프로세스의 결과물 (도면, BOM, 사양서, 시험 계획, KC 목록, FMEA 등) |
| ECR (Engineering Change Request) | 설계 변경 요청서 |
| ECN (Engineering Change Notice) | 설계 변경 통보서 |
| 게이트 (Gate) | 설계/개발 단계 간 진행 여부를 결정하는 공식 검토 포인트 |
| DR (Design Review) | 설계 검토 — 설계/개발 단계별 공식 검토 회의 |
| PDR (Preliminary Design Review) | 예비설계검토 |
| CDR (Critical Design Review) | 상세(치명)설계검토 |

| **영업팀장 (Kinh doanh)** | 고객 요구사항 전달; 고객 승인 요구사항 확인; 설계 변경 시 고객 커뮤니케이션 |

# 6. 설계/개발 기획 (8.3.1)

## 6.1 설계/개발 단계 및 게이트

설계/개발 프로세스는 APQP 5단계를 참조하여 다음과 같이 구성하며, 각 단계 완료 시 게이트 검토를 수행한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│             설계/개발 프로세스 흐름도 (Design & Development)            │
│             Design and Development Process Flow                      │
└─────────────────────────────────────────────────────────────────────┘

  고객 요구사항 / 신제품 기획
     │
     ▼
  ┌──────────────────────────────────────────┐
  │  Phase 0: 타당성 검토 (Feasibility)       │
  │  · 계약 검토 (QP-806)                    │
  │  · 기술 타당성 / 제조 가능성 평가          │
  │  · 리스크 예비 분석                       │
  │  · Go/No-Go 의사결정                     │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 0: 프로젝트 착수 승인
     │
  ┌──────────────────────────────────────────┐
  │  Phase 1: 기획 (Planning) — APQP 1단계   │
  │  · CFT 구성 및 킥오프                     │
  │  · 설계 입력 정의 (§7)                    │
  │  · 프로젝트 일정/WBS 수립                 │
  │  · 핵심특성(KC) 초기 식별                 │
  │  · DFMEA 초기 작성                        │
  │  · DVP&R 계획 수립                        │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 1: 기획 완료 검토 (Planning Review)
     │
  ┌──────────────────────────────────────────┐
  │  Phase 2: 설계/개발 실행 — APQP 2~3단계  │
  │  · 개념 설계 → 상세 설계                  │
  │  · DR-1: 개념설계 검토 (Concept DR)       │
  │  · PDR: 예비설계검토                      │
  │  · CDR: 상세설계검토                      │
  │  · 도면/BOM/사양서 작성                   │
  │  · DFMEA 완료, PFMEA 착수                │
  │  · DVP&R 상세화                           │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 2: 설계 동결 (Design Freeze)
     │
  ┌──────────────────────────────────────────┐
  │  Phase 3: 설계 검증 (Verification)        │
  │  · 시작품(Prototype) 제작                 │
  │  · DVP&R 실행 — 설계 검증 시험            │
  │  · 해석/시뮬레이션 결과 확인              │
  │  · 핵심특성 검증                          │
  │  · 설계 검증 결과 기록                    │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 3: 설계 검증 완료 (DV Complete)
     │
  ┌──────────────────────────────────────────┐
  │  Phase 4: 설계 확인 (Validation)          │
  │  · 양산 조건 시험 (사용 환경 모사)         │
  │  · 고객 요구사항 충족 확인                │
  │  · 규제/인증 요구사항 적합성 확인         │
  │  · 고객 승인 (해당 시)                    │
  │  · DVP&R 완료                             │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 4: 설계 확인 완료 (Validation Complete)
     │
  ┌──────────────────────────────────────────┐
  │  Phase 5: 양산 이관 — APQP 4~5단계       │
  │  · PPAP 제출 (해당 시)                    │
  │  · FAI 수행 (AS9102)                     │
  │  · 양산 전환 승인                         │
  │  · 초기 양산 모니터링                     │
  │  · 설계 기록 이관 및 보존                 │
  └──────────────────────────────────────────┘
     │
     ▼  ◆ Gate 5: 양산 이관 완료 (Production Transfer)

  ※ 전 단계 공통:
     - 각 Gate에서 Go/Conditional Go/No-Go 판정
     - 설계 변경 발생 시 → ECR/ECN 절차 (§10)
     - 형상관리 연계 (QP-803)
```

## 6.2 CFT (다기능팀) 구성

설계/개발 프로젝트 착수 시 다기능팀(CFT)을 구성한다.

| 구분 | 구성 | 역할 |
|------|------|------|
| 팀장 | 기술팀장 (개발부서장) | 프로젝트 총괄, 일정/자원/품질 관리 |
| 설계 담당 | 개발(Phát triển) 엔지니어 | 설계 실행, 도면/BOM/사양서 작성 |
| 품질 담당 | 품질(Chất lượng) 엔지니어 | FMEA 참여, 검사 계획 수립, FAI |
| 생산 기술 | 생산(Sản xuất) 엔지니어 | DFM 검토, 공정 설계, 양산 이관 |
| 금형 담당 | 금형(Khuôn) 엔지니어 | 금형/치공구 설계, 제작 일정 관리 |
| 구매 담당 | 구매(PUR) 담당자 | 부품/소재 조달, 공급업체 평가 |
| 고객 연락 | 영업(Kinh doanh) 담당자 | 고객 요구사항 확인, 승인 진행 |

CFT 구성 시 다음 사항을 고려한다:
- 프로젝트 규모 및 복잡도에 따라 구성원 조정 가능
- 외부 전문가(컨설턴트, 고객 대표) 참여 가능
- CFT 킥오프 회의 시 역할, 책임, 의사소통 채널을 명확히 정의
- 프로젝트 착수 시 **QF-807-01 (설계/개발 기획서)** 양식을 작성

## 6.3 설계/개발 기획 시 고려사항

설계/개발 기획 시 AS9100D 8.3.1에 따라 다음 사항을 결정한다:

| 고려사항 | 세부 내용 |

# 7. 설계/개발 입력 (8.3.3)

## 7.1 설계 입력 항목

설계/개발 착수 전 다음 입력 항목을 식별하고 문서화한다. 입력 항목 간 상충이 있을 경우 해결한다.

| **고객 특수 요구사항** | 고객별 품질 기준, 특수 검사 요구, 포장/표시 요구, 납품 조건 | 계약 검토 (QP-806) |

| **적용 규격/표준** | 산업 표준 (IPC, SAE 등), 재료 규격, 시험 규격 | 기술 데이터 패키지 |
| **생산 가능성** | DFM(Design for Manufacturing), DFA(Design for Assembly) 요구 | 생산부서, 금형부서 |
| **핵심특성 (KC) 후보** | 고객 지정 KC, 안전/기능 관련 잠정 KC | 고객 도면, 사양서 |

## 7.2 설계 입력 검토

- 설계 입력의 완전성, 명확성, 상호 모순 여부를 CFT가 검토한다
- 불명확한 요구사항은 고객 또는 이해관계자와 명확화 후 문서화한다
- 설계 입력 검토 결과를 **QF-807-02 (설계 입력 체크리스트)** 양식에 기록한다
- 설계 입력 확정 시 기술팀장이 승인하고, 이후 변경 시 ECR 절차를 따른다

# 8. 설계/개발 관리 (8.3.4)

## 8.1 설계 검토 (Design Review, DR)

> **CAR-017 시정조치**: 모든 설계 검토는 반드시 공식 회의로 수행하고, **QF-807-03 (설계 검토 회의록)** 양식에 기록한다. 구두 승인은 허용하지 않는다.

### 8.1.1 설계 검토 종류

| 검토 단계 | 시기 | 주관 | 참여자 | 검토 내용 |
|----------|------|------|--------|----------|
| DR-0: 타당성 검토 | Phase 0 완료 | 기술팀장 | CFT + QMR | 기술/제조/상업 타당성, Go/No-Go |
| DR-1: 개념설계 검토 | 개념설계 완료 | 기술팀장 | CFT | 설계 개념 적절성, 입력 충족 방향 |
| PDR: 예비설계검토 | 예비설계 완료 | 기술팀장 | CFT + QMR | 설계 방향, 인터페이스, KC 식별, 리스크 |
| CDR: 상세설계검토 | 상세설계 완료 | 기술팀장 | CFT + QMR | 설계 완결성, BOM/도면 적정성, FMEA |
| 최종 설계 검토 | 설계 동결 전 | 기술팀장 | CFT + QMR | 설계 동결 가부, 검증 계획 적정성 |

### 8.1.2 설계 검토 수행 절차

1. 기술팀장이 검토 의제, 참석자, 일정을 사전 통보 (최소 3영업일 전)
2. 해당 단계 설계 출력물을 사전 배포
3. 검토 회의에서 설계 적절성, 충분성, 효과성을 평가
4. 검토 결과 및 조치사항을 **QF-807-03**에 기록
5. 조치사항 담당자, 완료 기한을 지정
6. 조치사항 완료 후 기술팀장이 확인 서명
7. 게이트 판정: **Go / Conditional Go / No-Go**

### 8.1.3 설계 검토 참여자 요건

- 검토 대상 설계 단계에 관련된 기능 대표자가 반드시 참여
- 검토자는 설계 실행 담당자와 독립적인 인원을 포함
- 고객 대표의 참여가 요구되는 경우 사전 조율

## 8.2 설계 검증 (Design Verification, DV)

설계 검증은 설계/개발 출력이 설계/개발 입력 요구사항을 충족하는지 확인하는 활동이다.

> **AS9100D 요구사항**: 설계 검증(Verification)과 설계 확인(Validation)은 반드시 **별도의 활동**으로 수행한다 (8.3.4 Note 참조).

### 8.2.1 설계 검증 방법

| 방법 | 적용 예시 | 기록 |
|------|----------|------|
| 시험/계측 (Test) | 전기적 시험, 환경 시험, 수명 시험, 내구성 시험 | 시험 보고서 |
| 해석/시뮬레이션 (Analysis) | FEA, 열 해석, 회로 시뮬레이션, 공차 분석 | 해석 보고서 |
| 검사 (Inspection) | 치수 측정, 외관 검사, 비파괴 검사 | 검사 성적서 |
| 시연 (Demonstration) | 기능 시연, 조립성 확인, 작동 시연 | 시연 기록 |
| 설계 비교 (Comparison) | 유사 실적 제품과 비교 분석 | 비교 분석서 |

### 8.2.2 DVP&R (설계 검증 계획 및 보고서) 관리

> **CAR-017 시정조치**: DVP&R을 모든 설계/개발 프로젝트에 필수 적용한다.

DVP&R은 **QF-807-04 (DVP&R 양식)** 을 사용하여 다음 내용을 포함한다:

| DVP&R 항목 | 내용 |
|-----------|------|
| 설계 요구사항 | 검증 대상 요구사항 (설계 입력에서 도출) |
| 검증 방법 | 시험 / 해석 / 검사 / 시연 / 비교 중 선택 |
| 판정 기준 | 합격/불합격 기준 (정량적 기준 우선) |
| 시료 수량/조건 | 시험 시료 수, 시험 조건 (온도, 습도, 진동 등) |
| 담당자 | 검증 활동 수행 담당자 |
| 계획 일정 | 검증 활동 계획 완료일 |
| 결과 | 합격 / 불합격 / 조건부 합격 |
| 비고 | 불합격 시 조치사항, 재시험 계획 |

### 8.2.3 설계 검증 결과 처리

- **합격**: DVP&R에 결과 기록, 다음 단계 진행
- **불합격**: 근본원인 분석 → 설계 수정 → 재검증 (결과를 DVP&R에 추가 기록)
- **조건부 합격**: 조건 사항 및 후속 조치를 명시하여 기록, 제한적 단계 진행 허용

## 8.3 설계 확인 (Design Validation)

설계 확인은 최종 제품이 의도된 사용 또는 특정 용도의 요구사항을 충족하는지 확인하는 활동이다.

### 8.3.1 설계 확인 요건

| 항목 | 요구사항 |
|------|---------|
| 수행 시기 | 설계 검증 완료 후, 양산 이관 전 |
| 수행 조건 | 실제 사용 조건 또는 이를 모사한 조건에서 수행 |
| 독립성 | 설계 실행 담당자와 독립적인 인원이 주관 (품질팀장 주관) |
| 고객 참여 | 고객 요구 시 또는 계약 요구 시 고객 참여/입회 |
| 규제 적합 | 감항성, 안전 관련 규제 요구사항 적합성 확인 포함 |

### 8.3.2 설계 확인 활동

- 양산 조건 모사 시험 (생산 장비, 양산 재료 사용)
- 환경 시험 (사용 환경 조건에서의 기능/성능 확인)
- 규제/인증 시험 (감항성, EMC, 환경 규제 등)
- 고객 승인 시험 (고객 시험 사양에 따른 시험)
- DVP&R의 확인(Validation) 항목 실행 및 결과 기록

### 8.3.3 설계 확인과 검증의 분리

> **항공우주 필수 요구**: 설계 확인(Validation)은 설계 검증(Verification)과 반드시 별도의 활동으로 수행한다.

| 구분 | 설계 검증 (Verification) | 설계 확인 (Validation) |
|------|------------------------|---------------------|
| 질문 | "올바르게 만들었는가?" | "올바른 것을 만들었는가?" |
| 기준 | 설계 입력 요구사항 | 의도된 용도/사용 요구사항 |
| 주관 | 기술팀장 (개발부서) | 품질팀장 (독립적) |
| 시기 | Phase 3 (설계 검증 단계) | Phase 4 (설계 확인 단계) |
| 조건 | 시험실/해석 환경 | 실제 사용 조건 또는 모사 조건 |
| 기록 | DVP&R — Verification 섹션 | DVP&R — Validation 섹션 |

## 8.4 핵심특성 (Key Characteristics, KC) 관리

### 8.4.1 핵심특성 식별

다음 기준에 해당하는 특성을 핵심특성(KC)으로 식별한다:

| 기준 | 설명 |
|------|------|
| 안전 핵심 | 안전/감항성에 직접 영향을 미치는 특성 |
| 기능 핵심 | 제품의 핵심 기능/성능에 결정적 영향을 미치는 특성 |
| 고객 지정 | 고객이 핵심특성으로 지정한 특성 |
| 공정 핵심 | 변동 시 제품 적합성에 중대한 영향을 미치는 공정 파라미터 |

### 8.4.2 핵심특성 관리 프로세스

1. 설계 입력 단계에서 KC 후보를 식별
2. 설계 검토(DR) 시 KC를 확정하고 **QF-807-05 (핵심특성 목록)** 에 등록
3. KC를 도면, BOM, 사양서에 KC 기호(◇, △ 등)로 표기
4. DFMEA에서 KC 관련 고장 모드를 분석하고 관리 방안 수립
5. DVP&R에 KC 검증/확인 항목을 포함
6. 양산 시 KC에 대한 SPC(통계적 공정 관리) 또는 전수 검사 적용
7. KC 변경 시 ECR/ECN 절차 및 고객 승인 필요

# 9. 설계/개발 출력 (8.3.5)

## 9.1 필수 출력 항목

설계/개발 출력은 입력 요구사항에 대하여 검증 가능한 형태로 작성하며, 다음 항목을 포함한다:

| 출력 항목 | 내용 | 양식/형식 |
|----------|------|----------|
| 설계 도면 | 부품도, 조립도, 배치도, 회로도 (2D/3D) | CAD 파일 + PDF |
| BOM (자재명세서) | 부품 목록, 사양, 수량, 공급업체 | QF-807-06 또는 ERP |
| 제품 사양서 | 기능/성능 사양, 재료 사양, 환경 사양 | QF-807-07 |
| 시험 계획 / DVP&R | 설계 검증/확인 계획 및 결과 | QF-807-04 |
| 핵심특성 목록 | KC 목록, 관리 기준, 검사 방법 | QF-807-05 |
| DFMEA | 설계 고장 모드 영향 분석 | QP-802 양식 참조 |
| 검사 기준서 | 수입/공정/출하 검사 기준 | QF-807-08 |
| 포장 사양 | 포장 방법, 표시, 취급 주의사항 | 고객 요구 시 |
| 소프트웨어 문서 | 소프트웨어 요구사항, 설계서, 시험 계획 (해당 시) | 프로젝트별 정의 |

## 9.2 설계 출력 요구사항

AS9100D 8.3.5에 따라 설계/개발 출력은 다음을 충족한다:

- 설계/개발 입력 요구사항을 충족한다
- 후속 프로세스(구매, 생산, 서비스 제공)에 적절한 정보를 제공한다
- 모니터링 및 측정 요구사항과 합격 기준을 포함하거나 인용한다
- 의도된 목적과 안전/적절한 사용에 필수적인 제품 특성을 규정한다
- 핵심특성(KC)을 식별하고 관리 방법을 규정한다

## 9.3 설계 출력 승인

- 설계 출력은 기술팀장이 검토하고 QMR이 승인한다
- 승인된 설계 출력은 형상관리 대상으로 기준선(Baseline)에 포함 (QP-803)
- 설계 출력의 배포 및 관리는 문서화된 정보 관리 절차(QP-706)를 따른다

# 10. 설계/개발 변경 (8.3.6)

## 10.1 설계 변경 관리 프로세스

```
┌─────────────────────────────────────────────────────────────────┐
│              설계/개발 변경 관리 프로세스 흐름도                     │
│           Design/Development Change Management Flow               │
└─────────────────────────────────────────────────────────────────┘

  변경 필요 발생 (설계 오류, 고객 요구, 개선, 부적합)
     │
     ▼
  ① ECR (설계 변경 요청) 작성
     │  · 변경 사유, 변경 내용, 영향 범위 기술
     │  · 요청자 → 기술팀장 제출
     │
     ▼
  ② 영향 분석 (Impact Analysis)
     │  · 기술적 영향: 기능, 성능, 안전, KC 영향
     │  · 생산 영향: 공정, 금형, 재공품, 재고
     │  · 원가 영향: 부품비, 공정비, 금형비
     │  · 일정 영향: 납기, 생산 계획
     │  · 형상관리 영향: 기준선, CI 변경
     │
     ▼
  ③ CCB 심의/승인 (QP-803 연계)
     │  · CCB에서 변경 타당성 심의
     │  · 고객 승인 필요 여부 판단
     │
     ├──→ [반려] → ECR 보완 또는 종결
     │
     ▼
  ④ 고객 승인 (해당 시)
     │  · 고객 통보 및 승인 획득
     │  · 고객 승인 기록 보존
     │
     ▼
  ⑤ ECN (설계 변경 통보) 발행
     │  · 변경 실행 지시
     │  · 관련 부서 배포
     │
     ▼
  ⑥ 변경 실행
     │  · 도면/BOM/사양서 개정
     │  · 금형/지그 변경 (해당 시)
     │  · 생산 전환점 결정 (Effectivity)
     │  · 재공품/재고 처리 결정
     │
     ▼
  ⑦ 변경 검증
     │  · 변경된 설계의 재검증 (DVP&R 갱신)
     │  · 형상 기준선 갱신 (QP-803)
     │  · 변경 이력 기록
     │
     ▼
  ⑧ 변경 완료 및 기록 보존
     · ECR/ECN 이력 보존
     · 형상관리 기록 갱신
```

## 10.2 설계 변경 분류

| 분류 | 기준 | 승인 권한 | 고객 통보 |
|------|------|----------|----------|
| Class I (중대 변경) | 기능/성능/안전/적합성에 영향 | CCB + QMR + 고객 승인 | 필수 |
| Class II (경미 변경) | 기능/성능에 영향 없는 변경 | CCB + 기술팀장 | 고객 요구 시 |
| Class III (행정적 변경) | 오기 정정, 서식 변경 등 | 기술팀장 | 불필요 |

## 10.3 변경 시 재검증/재확인 요구

- Class I 변경: 변경 범위에 해당하는 설계 검증 및 확인 활동 재수행
- Class II 변경: 변경 범위에 해당하는 설계 검증 재수행
- 핵심특성(KC) 관련 변경: 고객 사전 승인 필수, DVP&R 갱신

## 10.4 형상관리 연계

- 모든 설계 변경은 QP-803(형상관리 절차)에 따라 CCB 심의를 거친다
- 변경 승인 후 해당 형상 항목의 기준선을 갱신한다
- ECR/ECN 번호와 형상 항목(CI) 번호의 추적 관계를 유지한다

# 11. 항공우주 특수 요구사항

## 11.1 설계 검증과 확인의 분리 (AS9100D 8.3.4)

- 설계 검증(Verification)과 설계 확인(Validation)은 별도의 활동으로 기획, 수행, 기록한다
- 검증은 기술팀(개발부서) 주관, 확인은 품질팀 주관으로 독립성을 보장한다
- DVP&R에서 Verification 항목과 Validation 항목을 구분하여 관리한다
- 각 활동의 완료 기준, 담당자, 일정을 별도로 정의한다

## 11.2 핵심특성 (Key Characteristics) 관리

- 모든 항공우주 제품의 설계 시 핵심특성(KC)을 식별하고 문서화한다
- KC는 도면에 특수 기호(◇)로 표기하고, 핵심특성 목록(QF-807-05)에 등록한다
- KC에 대해 DFMEA에서 고장 모드를 분석하고 탐지/예방 관리 방안을 수립한다
- KC 관련 변경 시 고객 사전 승인을 획득한다
- 양산 시 KC에 대해 SPC 또는 강화된 검사를 적용한다

## 11.3 DVP&R 의무 적용

- 모든 설계/개발 프로젝트에 DVP&R(QF-807-04)을 작성한다
- DVP&R은 설계 기획 단계에서 초기 작성하고, 설계 진행에 따라 상세화한다
- 검증/확인 결과는 DVP&R에 실시간으로 기록하고 관리한다
- DVP&R 완료는 양산 이관의 필수 전제조건이다

## 11.4 고객 승인 및 통보

다음 경우 고객에게 사전 통보 및 승인을 획득한다:

| 상황 | 조치 |
|------|------|
| 신규 설계 결과 | 고객 설계 승인 (해당 계약 조건에 따라) |
| 핵심특성(KC) 관련 변경 | 고객 사전 승인 필수 |
| Class I 설계 변경 | 고객 사전 승인 필수 |
| 재료/공정 변경 | 고객 통보 (계약 조건에 따라 승인 필요) |
| 공급업체 변경 | 고객 통보 (계약 조건에 따라 승인 필요) |
| 설계 이관 (생산지 변경) | 고객 사전 승인 필수 |

## 11.5 APQP/PPAP 연계

- Phase 별 APQP 산출물을 관리하고, 설계/개발 기획서(QF-807-01)에 해당 APQP 단계를 명시한다
- 고객 요구 시 PPAP 제출 패키지를 준비한다 (도면, BOM, FMEA, DVP&R, FAI, 관리 계획서 등)
- PPAP 승인 등급(Level 1~5)은 고객 지정에 따른다

## 11.6 설계 시 안전 고려사항

- 안전 핵심 항목에 대해 QP-804(제품/서비스 안전 관리)와 연계하여 관리한다
- 안전 관련 설계 변경은 반드시 안전성 영향 평가를 수행한다
- 안전 핵심 특성은 핵심특성 목록에 별도 구분하여 관리한다 (안전 KC)
